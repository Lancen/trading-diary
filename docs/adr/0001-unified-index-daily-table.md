# ADR-0001: 指数日线分表存储

## 状态

已接受（修订）

## 日期

2026-05-24

## 语境

项目需要存储三类指数的日线数据：
- **宽基指数**（MARKET）：上证指数、沪深300、中证500 等，来源 `stock_zh_index_daily`
- **行业指数**（INDUSTRY）：银行、半导体 等 ~446 个，来源 `stock_board_industry_index_ths`
- **概念指数**（CONCEPT）：ChatGPT、新能源车 等 ~400+ 个，来源 `stock_board_concept_index_ths`

现有 `index_daily` 表设计给宽基指数使用，`index_code VARCHAR(10)` 存标准交易所代码（如 `sh000001`），唯一键 `(index_code, trade_date)`。

行业/概念指数**没有标准交易所代码**，数量远超宽基（446+400 vs 8），且标识方式不同（按名称而非代码）。

## 决策

分两张表存储：
- **`index_daily`**：保持不变，仅存宽基指数（MARKET）
- **`sector_index_daily`**：新增，存行业/概念指数，`sector_type` 区分 INDUSTRY/CONCEPT

采集拆为 3 个独立类型：`MARKET_INDEX_DAILY`、`INDUSTRY_INDEX_DAILY`、`CONCEPT_INDEX_DAILY`，各自独立触发、互不影响。

## 理由

### 选择分表的原因

1. **标识体系不同**：宽基指数有标准交易所代码（`sh000001`），行业/概念指数无标准代码，只能用名称或自生成ID标识，混入会污染 `index_code` 字段语义
2. **数量级差异**：宽基 5-8 个 vs 行业/概念 850+ 个，混入后行业数据占 99%，宽基查询必须带 `index_type` 过滤
3. **唯一键冲突**：`index_daily` 现有唯一键 `(index_code, trade_date)` 无法区分行业和概念的同名情况，需加 `index_type` 才能解决
4. **查询隔离**：宽基指数查询（高频、大盘概览）和行业/概念查询（板块分析）是不同场景，分表避免互相干扰
5. **前端组件可复用**：K线+两融叠加图组件接收统一的 OHLCV 数据格式，不关心数据来自哪张表，分表不影响组件复用

### 选择行业/概念合表（而非三表）的原因

1. **数据结构完全一致**：行业和概念指数返回相同的 7 字段，无结构差异
2. **查询模式相同**：都是"按名称查K线+板块两融"，代码逻辑可复用
3. **`sector_type` 区分即可**：行业和概念不会同名，`sector_type` 过滤足够

### 选择 3 个独立采集类型的原因

1. **来源接口不同**：三个 AKShare 接口参数格式不同，无法统一调用
2. **失败隔离**：行业接口失败不影响宽基采集
3. **频率灵活**：宽基每日必采，行业/概念可按需降低频率

### 被否决的方案

| 方案 | 否决理由 |
|------|---------|
| 统一表 `index_daily` + `index_type` | `index_code` 语义被污染，唯一键需改造，宽基查询必须带过滤条件，行业数据占99%干扰主查询 |
| 三张表（`index_daily` / `industry_index_daily` / `concept_index_daily`） | 行业和概念数据结构、查询模式完全一致，分两表无必要，多一张表多一份维护 |

## 后果

- `index_daily` 表保持不变，无需迁移
- 新增 `sector_index_daily` 表，含 `sector_type` + `sector_name` 字段
- 前端图表组件需根据数据来源选择查询不同API，但组件内部渲染逻辑统一
- 未来新增板块类型（如申万行业指数）只需新增 `sector_type` 枚举值
