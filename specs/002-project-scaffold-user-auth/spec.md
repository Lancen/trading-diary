# Feature Specification: 项目脚手架 + 用户认证体系

**Feature Branch**: `002-project-scaffold-user-auth`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: 搭建项目脚手架和用户认证体系。管理员通过种子数据初始化（用户名 admin，密码从环境变量配置）。系统支持账号密码登录，验证成功后签发短期访问令牌和长期刷新令牌。令牌过期后可刷新，登出后刷新令牌失效。所有密码加密存储。预置 ADMIN 和 USER 两种角色用于权限控制。开发阶段启动时自动以管理员身份登录，开发者无需手动输入密码即可访问所有接口。生产模式下自动登录失效，所有接口必须携带有效令牌。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理员登录 (Priority: P1)

管理员通过用户名和密码登录系统，获取访问凭证，从而使用系统功能。

**Why this priority**: 认证是所有业务功能的前置条件。没有认证，数据归属隔离和权限控制无法实施。

**Independent Test**: 调用登录接口，提交正确凭证后获得令牌，使用该令牌访问受保护资源成功。仅需一个管理员账号即可完成测试。

**Acceptance Scenarios**:

1. **Given** 系统已初始化管理员账号，**When** 管理员提交正确的用户名和密码，**Then** 系统返回短期访问凭证和长期刷新凭证及凭证有效期。
2. **Given** 系统已初始化管理员账号，**When** 管理员提交错误的密码，**Then** 系统返回认证失败提示，不签发任何凭证。
3. **Given** 管理员账号已被禁用，**When** 该管理员尝试登录，**Then** 系统返回账号已禁用的提示。
4. **Given** 管理员持有有效访问凭证，**When** 管理员访问受保护的资源，**Then** 系统正常返回资源内容。
5. **Given** 管理员未登录或访问凭证已过期，**When** 管理员访问受保护的资源，**Then** 系统返回未认证提示。

---

### User Story 2 - 凭证刷新与安全登出 (Priority: P2)

管理员在访问凭证过期后使用刷新凭证获取新的访问凭证；主动登出后所有凭证失效，防止凭证泄漏后被滥用。

**Why this priority**: 凭证刷新保证使用体验的连续性（避免频繁重新登录）。安全登出是信息安全的基本要求。

**Independent Test**: 登录后获取凭证对，使用刷新凭证换取新访问凭证，调用登出后验证原刷新凭证失效。

**Acceptance Scenarios**:

1. **Given** 管理员持有有效的刷新凭证，**When** 管理员请求刷新访问凭证，**Then** 系统返回新的访问凭证和新的刷新凭证，原刷新凭证立即失效。
2. **Given** 管理员持有的刷新凭证已失效或已被撤销，**When** 管理员请求刷新，**Then** 系统返回认证失败提示。
3. **Given** 管理员已登录，**When** 管理员主动登出，**Then** 当前刷新凭证被标记为失效，后续使用该凭证的任何请求均被拒绝。

---

### User Story 3 - 开发环境自动登录 (Priority: P1)

开发阶段，开发者打开任何页面即自动以管理员身份登录，无需每次手动输入账号密码。

**Why this priority**: 开发效率的基础。与登录功能并列 P1，因为它是开发体验的前置条件——每次重启应用都需要手动登录会显著降低开发迭代速度。

**Independent Test**: 在开发模式下访问任意受保护资源（不携带任何凭证），验证系统以管理员身份正常返回数据。切换到生产模式后重复此操作，验证系统拒绝访问。

**Acceptance Scenarios**:

1. **Given** 系统运行在开发模式下，**When** 开发者访问任意受保护资源（无需携带凭证），**Then** 系统自动以管理员身份处理请求，正常返回内容。
2. **Given** 系统运行在生产模式下，**When** 开发者访问受保护资源（无需携带凭证），**Then** 系统返回未认证提示——自动登录功能不生效。

---

### Edge Cases

- 系统首次启动时自动初始化管理员账号，重复启动时如何避免创建重复账号？（应当检测管理员是否已存在，存在则跳过，保证操作幂等）
- 访问凭证在请求处理过程中刚好过期怎么办？（应当在收到凭证过期提示后自动尝试刷新，刷新失败再要求重新登录）
- 刷新凭证即将到期但管理员仍在操作怎么办？（应在凭证过期前主动刷新，预留缓冲时间）
- 开发模式下自动登录时，管理员账号在数据库中不存在怎么办？（启动时应校验管理员账号存在性，不存在则拒绝启用自动登录）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供账号密码登录功能，验证成功后签发短期访问凭证和长期刷新凭证
- **FR-002**: 系统 MUST 对所有存储的密码进行加密处理
- **FR-003**: 系统 MUST 在每次访问受保护资源时验证访问凭证的合法性和时效性
- **FR-004**: 系统 MUST 提供凭证刷新功能，接受有效刷新凭证并返回新的凭证对，同时使旧刷新凭证失效
- **FR-005**: 系统 MUST 提供登出功能，使当前刷新凭证立即失效
- **FR-006**: 系统 MUST 在开发模式下自动识别开发者身份，无需手动登录即可访问所有资源
- **FR-007**: 系统 MUST 在生产模式下禁用自动登录，所有受保护资源必须携带有效凭证
- **FR-008**: 系统 MUST 在首次启动时自动创建管理员账号，密码从环境配置读取
- **FR-009**: 系统 MUST 区分受保护资源和公开资源（登录、刷新凭证等接口允许匿名访问）
- **FR-010**: 系统 MUST 对所有认证失败场景返回统一结构的错误信息
- **FR-011**: 系统 MUST 预置管理员和普通用户两种角色，用于权限区分
- **FR-012**: 系统 MUST 在重复初始化时幂等处理——已存在的管理员账号和角色不重复创建也不报错

### Key Entities

- **用户 (User)**: 系统使用者。拥有用户名（唯一标识）、加密密码、显示名称、状态（正常/禁用）。至少关联一个角色。
- **角色 (Role)**: 权限集合。拥有角色编码（唯一标识，如 ADMIN、USER）和角色名称。通过关联关系与用户连接。
- **访问凭证 (Access Token)**: 短期有效的身份凭证，承载用户身份和角色信息，用于每次请求的权限验证。
- **刷新凭证 (Refresh Token)**: 长期有效的凭证，唯一用途是换取新的访问凭证，不用于日常请求。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 开发者在开发模式下启动系统后，无需任何手动认证操作即可访问任意功能
- **SC-002**: 管理员使用正确账号密码登录，2 秒内获得访问凭证
- **SC-003**: 凭证验证过程对每次请求的额外耗时不超过 50 毫秒
- **SC-004**: 系统首次启动自动创建管理员账号，重复启动不产生重复数据或错误
- **SC-005**: 所有认证失败场景（错误密码、过期凭证、禁用账号）返回结构一致的错误信息
- **SC-006**: 生产模式下，未携带有效凭证的请求 100% 被拒绝

## Assumptions

- 管理员初始密码通过环境变量配置，开发环境有合理的默认值
- 开发模式和生产模式通过运行时配置区分，不依赖代码修改
- 本功能不包含用户自主注册——管理员账号通过系统初始化自动创建
- 本功能不包含权限管理后台界面，仅建立角色体系框架
- 本功能不包含请求频率限制、容器化部署、端到端自动化测试（这些在后续阶段实现）
- 管理员用户名固定为 `admin`
