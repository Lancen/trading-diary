# Feature Specification: 项目脚手架 + 用户体系

**Feature Branch**: `001-project-scaffold-user-auth`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: 搭建项目骨架和认证体系——Spring Boot 脚手架、JWT 无状态认证、RBAC 角色权限、管理员种子数据、开发环境自动登录。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理员登录（生产环境）(Priority: P1)

管理员通过用户名和密码登录系统，获取访问凭证，进入管理后台。

**Why this priority**: 认证是所有业务功能的前置条件。没有认证，数据归属隔离、权限控制都无从谈起。

**Independent Test**: 通过调用登录接口并验证返回的访问令牌来独立测试。无需依赖其他功能模块。

**Acceptance Scenarios**:

1. **Given** 系统中存在管理员账号，**When** 用户提交正确的用户名和密码，**Then** 系统返回有效的访问令牌（短期）和刷新令牌（长期），以及令牌过期时间。
2. **Given** 系统中存在管理员账号，**When** 用户提交错误的密码，**Then** 系统返回认证失败错误，不返回任何令牌。
3. **Given** 管理员账号被禁用，**When** 用户提交正确的用户名和密码，**Then** 系统返回账号已禁用错误。
4. **Given** 用户已登录，**When** 用户携带有效令牌访问受保护资源，**Then** 系统正常返回资源。
5. **Given** 用户未登录或令牌过期，**When** 用户访问受保护资源，**Then** 系统返回未认证错误。

---

### User Story 2 - 令牌刷新与登出 (Priority: P2)

用户在访问令牌过期后使用刷新令牌获取新的访问令牌，以及主动登出使刷新令牌失效。

**Why this priority**: 令牌刷新是保持用户体验连续性的关键，避免频繁重新登录。登出是安全基本要求。

**Independent Test**: 通过登录获取令牌，使用刷新令牌换取新访问令牌，调用登出后验证原刷新令牌失效来独立测试。

**Acceptance Scenarios**:

1. **Given** 用户持有有效的刷新令牌，**When** 用户请求刷新访问令牌，**Then** 系统返回新的访问令牌和新的刷新令牌，旧的刷新令牌失效（令牌轮换）。
2. **Given** 用户持有已失效或被撤销的刷新令牌，**When** 用户请求刷新访问令牌，**Then** 系统返回认证失败错误。
3. **Given** 用户已登录，**When** 用户主动登出，**Then** 系统使当前刷新令牌失效，后续刷新请求被拒绝。

---

### User Story 3 - 开发环境自动登录 (Priority: P1)

开发阶段（dev 模式），开发者打开任何页面自动以管理员身份登录，无需手动输入账号密码。

**Why this priority**: 开发阶段的效率基础。没有自动登录，每次重启应用都需要手动登录，严重影响开发效率。与 P1 登录并列因为它是开发体验的前置条件。

**Independent Test**: 在 dev 模式下直接访问任意受保护接口，验证无需 Authorization 头即可以管理员身份获取数据。

**Acceptance Scenarios**:

1. **Given** 系统运行在 dev 模式下，**When** 开发者访问任意受保护的页面或接口（无需携带 Authorization 头），**Then** 系统自动以管理员身份处理请求。
2. **Given** 系统运行在 prod 模式下，**When** 开发者访问受保护接口（无需携带 Authorization 头），**Then** 系统返回未认证错误——自动登录不生效。

---

### Edge Cases

- 系统启动时种子数据中的管理员账号已存在（重复启动场景），如何处理？（应幂等：检测到 admin 已存在则跳过插入）
- 访问令牌在并发请求期间过期如何处理？（前端应在收到 401 时自动尝试刷新，刷新失败再跳转登录）
- 刷新令牌即将到期时用户仍在活跃操作如何处理？（前端在令牌过期前主动刷新，预留缓冲时间）
- dev 模式自动登录的用户在数据库中不存在如何处理？（启动时校验：admin 用户必须存在，否则打印错误日志并阻止自动登录过滤器生效）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供账号密码登录接口，验证成功后返回访问令牌和刷新令牌
- **FR-002**: 系统 MUST 使用 BCrypt 加密存储所有用户密码
- **FR-003**: 系统 MUST 在每次请求时验证访问令牌的签名和有效期
- **FR-004**: 系统 MUST 提供令牌刷新接口，接受有效刷新令牌并返回新令牌对（令牌轮换）
- **FR-005**: 系统 MUST 提供登出接口，使当前刷新令牌失效
- **FR-006**: 系统 MUST 在 dev 模式下提供自动登录机制，开发者无需手动认证
- **FR-007**: 系统 MUST 在 prod 模式下禁用自动登录，所有接口需要有效令牌
- **FR-008**: 系统 MUST 通过数据库种子数据初始化管理员账号，密码通过环境变量配置
- **FR-009**: 系统 MUST 支持 RBAC 角色体系，预置 ADMIN 和 USER 两种角色
- **FR-010**: 系统 MUST 区分受保护端点和匿名端点（登录、刷新令牌接口允许匿名访问）
- **FR-011**: 系统 MUST 对所有认证失败返回统一格式的错误响应
- **FR-012**: 系统 MUST 在种子数据重复运行时幂等处理（已存在的管理员/角色不重复创建）

### Key Entities

- **用户 (User)**: 系统使用者。属性：用户名（唯一）、加密密码、昵称、状态（正常/禁用）。至少拥有一个角色。
- **角色 (Role)**: 权限的集合。属性：角色编码（唯一，如 ADMIN、USER）、角色名称。通过关联表与用户建立多对多关系。
- **权限 (Permission)**: 具体的操作许可。属性：权限编码（如 `trade:read`）。本阶段建表但不填充数据，为后续阶段预留。
- **访问令牌 (Access Token)**: 短期有效的认证凭证（15 分钟），承载用户身份和角色信息。
- **刷新令牌 (Refresh Token)**: 长期有效的凭证（7 天），仅用于换取新的访问令牌。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 开发者在 dev 模式下启动应用后，无需任何手动认证操作即可直接调用任意 API 接口
- **SC-002**: 用户使用正确账号密码登录，在 2 秒内获得访问令牌
- **SC-003**: 令牌验证对每次 API 请求的额外延迟不超过 50 毫秒
- **SC-004**: 管理员种子数据在首次启动时自动创建，重复启动不会产生重复数据或错误
- **SC-005**: 所有认证失败场景（错误密码、过期令牌、禁用账号）返回一致结构的错误响应
- **SC-006**: prod 模式下，未携带令牌的请求 100% 被拒绝

## Assumptions

- 管理员初始密码通过环境变量 `ADMIN_INIT_PASSWORD` 传入，dev 模式下有默认值
- dev 模式通过应用运行 profile 区分，由部署配置控制，不依赖代码修改
- 前端在 dev 模式下配合后端自动登录，无需跳转登录页
- 本阶段不包含用户注册功能，仅通过种子数据创建管理员
- 本阶段不包含权限管理后台界面，仅建表和 RBAC 骨架
- 本阶段不包含 API 限流、Docker 部署、E2E 测试（后续阶段实现）
- 管理员用户名固定为 `admin`
