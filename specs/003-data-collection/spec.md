# 功能规范：数据采集层

**Feature Branch**: `003-data-collection`  
**创建日期**: 2026-05-13  
**状态**: 草稿  
**输入**: 用户描述: "管理员可查看各数据类的采集状态（成功/失败/运行中），手动触发采集或对缺失数据发起补采。系统定时自动从外部数据源采集以下数据：A 股全量股票基础信息（代码、名称、最新价、涨跌幅、PE/PB/总市值/流通市值等，保留每日快照）、股票日线行情（开盘/最高/最低/收盘/成交量/成交额）、东方财富行业板块分类及各行业成分股、东方财富概念板块分类及各概念成分股、A 股交易日历、两融标的列表（融资标的/融券标的/两者）、每日两融交易明细（融资余额/融资买入额/融资偿还额/融券余额/融券卖出量/融券余量等，按股票按交易日）。历史数据按周回填，支持断点续传。系统自动检测两融数据缺口，管理员可按周查看数据完整性并补采缺失交易日。所有采集任务失败时自动重试三次，失败信息在管理后台可见。"

## 用户场景与测试 *(必填)*

### User Story 1 - 定时自动采集数据 (Priority: P1)

管理员无需任何操作，系统在每个交易日自动从外部数据源采集最新数据，包括：股票基础信息快照、股票日线行情、行业板块分类及成分股、概念板块分类及成分股、A 股交易日历、两融标的列表、每日两融交易明细。采集完成后自动完成数据清洗入库。

**为什么是这个优先级**: 数据采集是整个系统的数据来源。没有每日自动采集，后续所有股票查询和两融分析功能都无法运行。这是系统存在的前提。

**独立测试**: 配置两个交易日（如周三和周四），在每个交易日的收盘时间后观察系统是否自动拉取了当天的全量数据并写入成功。无需前端页面即可通过数据库验证。

**验收场景**:

1. **假设** 当天为交易日 16:00 后，**当** 定时任务触发，**则** 系统自动采集全 A 股股票基础信息快照（含代码、名称、最新价、涨跌幅、PE、PB、总市值、流通市值等）并保留当天快照记录
2. **假设** 当天为交易日 16:00 后，**当** 定时任务触发，**则** 系统自动采集全 A 股的日线行情（开盘价、最高价、最低价、收盘价、成交量、成交额）
3. **假设** 当天为交易日 17:00 后，**当** 定时任务触发，**则** 系统自动采集东方财富行业板块列表、各行业成分股、概念板块列表、各概念成分股
4. **假设** 当天为交易日 18:00 后，**当** 定时任务触发，**则** 系统自动采集当日沪深两市两融交易明细（个股级别的融资余额、融资买入额、融资偿还额、融券余额、融券卖出量、融券偿还量、融券余量、两融总余额）
5. **假设** 系统首次启动，**当** 定时任务触发交易日历采集，**则** 自动拉取 A 股全部历史交易日列表（自 1990 年起）
6. **假设** 两融交易明细采集中出现新的股票代码，**当** 数据清洗时，**则** 该股票自动加入两融标的列表

### User Story 2 - 查看采集状态与数据完整性 (Priority: P2)

管理员登录管理后台后，可以查看所有数据类的最后一次采集状态（成功/失败/运行中），包括原始数据采集和数据清洗各自的状态、执行时间和记录数。对于两融数据，可以按周查看数据完整性——哪些周完整、哪些周存在缺失交易日，以及具体缺失哪些日期。

**为什么是这个优先级**: 管理员需要知道数据是否正常入库。没有状态可见性，数据出问题也无法发现。但状态查看本身不产生数据，优先级低于自动采集。

**独立测试**: 模拟一次成功采集和一次失败采集，打开管理后台页面，验证状态卡片是否正确显示成功/失败状态。在数据库中删除某两天的两融数据，打开完整性页面，验证是否正确标记缺口。

**验收场景**:

1. **假设** 当天数据采集全部成功，**当** 管理员打开采集状态页面，**则** 看到所有数据类的 FETCH 和 CLEANSE 状态均显示"成功"，并显示采集时间和记录数
2. **假设** 某类数据采集失败（如外部数据源不可用），**当** 管理员打开采集状态页面，**则** 该类数据显示红色"失败"标记，展开可看到失败原因
3. **假设** 某类数据正在采集中，**当** 管理员打开采集状态页面，**则** 该类数据显示"运行中"状态
4. **假设** 两融数据存在若干交易日缺失，**当** 管理员打开两融完整性页面并选择日期范围，**则** 看到按周展示的完整性表格：应采 X 天、已采 Y 天、缺口列出具体缺失日期
5. **假设** 两融数据完整无缺口，**当** 管理员查看任一周期，**则** 该周显示"完整"状态

### User Story 3 - 手动触发与补采 (Priority: P3)

管理员可以手动触发任意数据类的采集任务，也可以按日期范围对缺失数据发起补采。对于两融历史数据，补采按周分片执行，中断后可以从断点继续，不需要从头开始。

**为什么是这个优先级**: 手动触发和补采是运维兜底手段。定时采集正常情况下不需要人工干预，但一旦出现数据源故障、网络中断或历史数据缺失，没有补采能力就将导致数据永久缺失。这是防御性能力。

**独立测试**: 在管理后台对"股票基础信息"点手动触发，观察是否即刻发起一次采集并更新状态。对两融数据指定日期范围点补采，观察是否按周分片执行并逐步消除缺口。

**验收场景**:

1. **假设** 管理员想立即刷新某类数据，**当** 在管理后台点击"重新采集"按钮，**则** 系统立即发起该数据类型的采集任务，采集完成后页面更新为最新状态
2. **假设** 两融数据缺口页面显示某周缺失 3 天，**当** 管理员点击"补采缺失"，**则** 系统按周对该周缺失的 3 天逐一发起采集，补采完成后该周状态变为"完整"
3. **假设** 管理员要补采大范围历史数据（如 2024 年全年），**当** 选择日期范围并点击"开始补采"，**则** 系统按周分片逐步执行，管理员可以看到采集日志中每周的进度
4. **假设** 补采过程中系统重启，**当** 重启后，**则** 管理员可重新发起补采，已完成的周不会重复采集（断点续传）
5. **假设** 同一数据类型正在采集中（运行中状态），**当** 管理员再次点击手动触发，**则** 系统拒绝重复执行，提示"已在执行中"
6. **假设** 采集任务失败 3 次，**当** 管理员查看该数据类的状态，**则** 可以看到详细的失败原因描述

### User Story 4 - 行业与概念分类变更追踪 (Priority: P3)

系统自动检测股票行业分类和概念归属的变化（如某只股票从"电子"行业调入"计算机"行业），记录每一次变更的时间、类型（新增/移除）和涉及的板块。管理员可以追溯任意股票在任意时间段内的行业和概念分类历史。

**为什么是这个优先级**: 行业分类变化频率低（通常月度甚至季度调整），但后续按行业/概念聚合两融数据时必须知道某日某股属于哪个板块。这是分析准确性的保障。

**独立测试**: 假设某日某只股票的行业从"电子"变为"计算机"，当天采集后检查变更日志是否记录了该变化（股票代码 + 旧行业移除 + 新行业添加）。模拟历史数据聚合查询，验证是否使用了正确的历史分类。

**验收场景**:

1. **假设** 某只股票在本次采集中的行业归属与上次不同，**当** 数据清洗完成，**则** 行业关联表更新为最新，变更日志表记录原行业移除和新行业添加两条记录
2. **假设** 某只股票首次被采集到成分股中，**当** 数据清洗完成，**则** 变更日志记录"添加"操作
3. **假设** 股票行业/概念归属无变化，**当** 再次采集，**则** 不产生新的关联记录和变更日志

---

### 边界场景

- 外部数据源不可用时（AKTools 服务异常或数据源网站限流），采集任务应自动重试 3 次（递增间隔），3 次均失败后记录错误信息，管理后台红色标记。下次定时任务到点时重新尝试。
- 数据清洗过程中系统崩溃：清洗操作在单次事务内完成，失败回滚，不会出现部分数据写入、部分未写入的状态。
- 非交易日（周末/节假日）定时任务触发：交易日历维护了全部交易日列表。定时任务执行前检查当天是否为交易日，非交易日自动跳过。
- 历史回填数据量巨大（2010 年至今约 3900 个交易日）：按周分片执行，支持断点续传。优先回填最近的数据，再逐步往前回溯。
- 同一数据类型的定时采集和手动触发同时发生：系统拒绝重复执行，返回"已在执行中"。

## 功能需求 *(必填)*

### 功能需求

- **FR-001**: 系统 MUST 在每个交易日自动采集全 A 股股票基础信息（代码、名称、最新价、涨跌幅、涨跌额、成交量、成交额、换手率、量比、市盈率、市净率、总市值、流通市值），保留每日快照
- **FR-002**: 系统 MUST 在每个交易日自动采集全 A 股日线行情（开盘价、最高价、最低价、收盘价、成交量、成交额），收盘价使用前复权标准
- **FR-003**: 系统 MUST 在每个交易日自动采集东方财富行业板块列表及每个行业的成分股
- **FR-004**: 系统 MUST 在每个交易日自动采集东方财富概念板块列表及每个概念的成分股
- **FR-005**: 系统 MUST 自动采集并维护 A 股交易日历（全量历史交易日列表，支持增量更新）
- **FR-006**: 系统 MUST 在每个交易日自动采集沪深两市每日两融交易明细（个股级别的融资余额、融资买入额、融资偿还额、融券余额、融券卖出量、融券偿还量、融券余量、两融总余额）
- **FR-007**: 系统 MUST 自动从两融交易明细中提取并维护两融标的列表
- **FR-008**: 系统 MUST 在每次采集时先将原始数据保存，再执行清洗入库，两步各有独立状态记录
- **FR-009**: 系统 MUST 为每次采集和清洗任务记录执行状态（运行中/成功/失败）、开始时间、完成时间、记录数量和失败原因
- **FR-010**: 系统 MUST 在采集失败时自动重试 3 次（间隔递增），3 次均失败后标记为失败状态并记录详细错误信息
- **FR-011**: 管理员 MUST 能够在管理后台查看所有数据类的最新采集状态和最近的执行日志
- **FR-012**: 管理员 MUST 能够在管理后台手动触发任意数据类的采集任务
- **FR-013**: 管理员 MUST 能够在管理后台按日期范围发起补采任务
- **FR-014**: 系统 MUST 自动检测两融数据缺失的交易日（以交易日历为基准），并按周展示数据完整性
- **FR-015**: 管理员 MUST 能够按周查看两融数据完整性（应采天数、已采天数、具体缺失日期）
- **FR-016**: 历史数据回填 MUST 按周分片执行，每个周片独立记录采集状态，支持断点续传
- **FR-017**: 系统 MUST 拒绝同一数据类型的并发采集请求（定时任务和手动触发冲突时，后者返回"已在执行中"）
- **FR-018**: 行业和概念成分股的数据清洗 MUST 检测与上次快照的差异，仅写入变更部分
- **FR-019**: 系统 MUST 记录每只股票行业分类和概念归属的变更历史（包括变更类型：添加/移除，变更发现的日期）
- **FR-020**: 系统 MUST 将超过 30 天的原始数据归档为文件存储，并从数据库中删除已归档记录

### 关键实体 *(必填)*

- **采集任务**: 代表一次数据采集或清洗操作。包含数据类型、任务类型（原始采集/清洗入库）、执行状态、关联的交易日、按周回填时的周范围、执行时间、记录数和错误信息。
- **原始数据**: 从外部数据源拉取的未经处理的 JSON 数据。关联到具体的采集任务，包含数据类型、交易日和采集时间。
- **交易日历**: A 股所有交易日日期列表。用于判断某天是否为交易日，以及作为两融数据完整性检测的基准。
- **股票基础信息快照**: 一只股票在某一天的基础信息快照（代码、名称、最新价、涨跌幅、涨跌额、成交量、成交额、换手率、量比、市盈率、市净率、总市值、流通市值）。同一只股票在不同日期保留独立记录。
- **股票日线行情**: 一只股票在某一天的日线 OHLCV 数据（开盘价、最高价、最低价、收盘价、成交量、成交额）。同一只股票在不同日期保留独立记录。
- **行业板块**: 东方财富行业分类下的板块定义（板块代码和名称）。
- **股票-行业关联**: 一只股票当前所属的行业板块。每只股票只属于一个行业（东方财富单分类模式），仅保留最新状态。
- **概念板块**: 东方财富概念分类下的板块定义（板块代码和名称）。
- **股票-概念关联**: 一只股票当前所属的概念板块。一只股票可属于多个概念，仅保留最新状态。
- **分类变更日志**: 记录股票行业或概念归属的每次变更。包含股票代码、分类类型（行业/概念）、板块代码、操作类型（添加/移除）和变更日期。
- **两融标的**: 可进行融资融券交易的股票列表。包含股票代码、名称、所属交易所、是否融资标的、是否融券标的和快照日期。
- **两融交易明细**: 一只两融标的在某一个交易日的两融交易数据。包含融资余额、融资买入额、融资偿还额、融券余额、融券卖出量、融券偿还量、融券余量、两融总余额。按股票、交易日、交易所唯一确定一条记录。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**: 每个交易日收盘后，6 类核心数据（股票基础信息、日线行情、行业板块、概念板块、两融标的、两融交易明细）在 30 分钟内全部完成采集和清洗入库
- **SC-002**: 采集任务失败后 3 次自动重试的完成时间不超过 30 秒（含递增间隔 2s+4s+8s=14s × 3 次）
- **SC-003**: 管理员在管理后台可以直观看到每类数据的采集状态，从进入页面到找到目标数据类状态不超过 10 秒
- **SC-004**: 两融数据完整性检测在指定日期范围内（如 2024 年全年）的缺口计算在 5 秒内完成并展示
- **SC-005**: 管理员从点击"补采缺失"到系统开始执行补采任务的响应时间不超过 2 秒
- **SC-006**: 行业/概念分类变更检测的准确率达到 100%（不遗漏任何成分股变化）
- **SC-007**: 首次历史全量回填（2014 年至今约 2900 个交易日 × 2000+ 只两融标的）在 2 小时内完成

## 假设条件

- 外部数据源（Akshare/AKTools）的 HTTP API 在交易日提供稳定的数据服务，非交易日不需要采集
- 东方财富行业/概念板块数据在每个交易日收盘后更新，当日采集到的即为最新分类
- 两融交易明细数据由交易所在每个交易日收盘后公布，通常 17:30 后可获取
- 管理员拥有系统的 ADMIN 角色，管理后台需要登录认证
- 当前版本不包含邮件/站内信等主动告警通知，失败信息通过管理后台页面被动查看
- 原始数据的归档文件存储在服务器本地文件系统，不涉及外部对象存储服务
- 子项目 1（项目脚手架 + 用户认证体系）已完成，系统具备用户认证和角色管理基础
- 股票日线行情的收盘价统一使用前复权标准（流通股本变化时向前调整历史价格）
